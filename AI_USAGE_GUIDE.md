# AI 辅助开发指南

## 为什么允许使用 AI？

在真实工作中，AI 工具已经成为提升效率的重要助手。我们希望你展示的是：

> **与 AI 有效协作的能力，而非闭卷答题能力。**

---

##  我们期望的 AI 使用方式

###  推荐做法

| 场景 | 示例 |
|------|------|
| **生成样板代码** | "帮我写一个 Spring Boot Controller 处理入库单创建" |
| **代码审查** | "这段事务处理代码有什么潜在问题？" |
| **调试辅助** | "这段 SQL 为什么查询很慢？" |
| **学习新技术** | "FastAPI 中如何做请求参数校验？" |
| **重构建议** | "这个 Service 方法太长，如何拆分更合理？" |
| **测试生成** | "为这个入库方法生成单元测试" |
| **文档生成** | "帮我生成这个接口的 API 文档" |

###  不推荐做法

| 做法 | 为什么不推荐 |
|------|------------|
| **全量复制粘贴不审查** | AI 生成的代码可能有逻辑错误、安全漏洞 |
| **不理解就提交** | 如果面试时你无法解释自己的代码，会被扣分 |
| **不验证边界条件** | AI 常忽略空值、并发、异常等边界情况 |
| **不管理 Prompt 上下文** | 大量无关代码喂给 AI 会降低输出质量 |

---

##  推荐的 AI 协作工作流

```
1. 理解需求
   └── 先自己读一遍任务，画出数据流

2. 与 AI 讨论方案
   └── "对于入库单创建，我打算用 @Transactional，你觉得有什么需要注意的？"

3. AI 生成初版代码
   └── 将相关的实体类、DTO、表结构作为 context 喂给 AI
   └── 明确要求：语言、框架、异常处理、日志

4. 审查 + 修改
   └── 检查事务边界是否正确
   └── 检查异常处理是否完善
   └── 手动补充 AI 遗漏的边界条件

5. 测试验证
   └── 运行、测试、修 bug
   └── 如有问题，将错误信息喂给 AI 辅助排查

6. 小步提交
   └── 每完成一个功能点，提交一次
   └── commit message 写清楚做了什么
```

##  Prompt 编写技巧

### 提供充分上下文

```
 差：写一个入库功能
 好：我正在用 Spring Boot 3 + JPA 开发 WMS 系统。
      以下是数据库表结构（粘贴 DDL）和现有 ProductService 代码（粘贴代码）。
      请帮我实现 InboundOrderService.batchCreate 方法，要求：
      1. 使用 @Transactional 保证一致性
      2. 入库单号格式 IN-YYYYMMDD-XXX
      3. 库存不足时抛出 BusinessException
      4. 加上必要的参数校验
```

### 分步骤提问

对于复杂功能，拆成多步：

```
第1步：先帮我设计入库单的数据结构和 API 接口定义
第2步：（审查通过后）根据这个设计生成 Service 实现
第3步：生成单元测试
```

### 利用 AI 做 Code Review

```
"Review 以下代码，从以下角度分析：事务安全、SQL 性能、空指针风险、并发问题"
```

##  我们如何评估 AI 使用

在 `NOTES.md` 中请如实记录：

1. 你使用了哪些 AI 工具？（ChatGPT / Copilot / Claude / 其他）
2. 举 1 个具体例子：AI 帮你解决了什么问题？
3. 举 1 个具体例子：AI 生成的代码有什么问题，你是如何发现和修复的？

我们不会因为你用了 AI 而扣分。相反，**善用 AI 且能批判性审视其输出**是加分项。
